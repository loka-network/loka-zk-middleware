# Loka ZK Middleware

A cloud API middleware microservice for **Zero-Knowledge (ZK) proof generation and verification**, built with Rust for maximum performance and security. Designed to enhance user privacy and security, with seamless integration into **Agentic payment flows**.

## Overview

Loka ZK Middleware provides a RESTful API service that enables applications to:

- **Generate ZK proofs** — Create cryptographic proofs that verify computations without revealing private inputs
- **Verify ZK proofs** — Validate proofs submitted by any party, ensuring correctness without accessing secrets
- **Enhance privacy in Agentic payments** — Enable AI agents to prove payment validity, amount ranges, and transaction splits without exposing sensitive financial data

### Why Zero-Knowledge Proofs for Agentic Payments?

As AI agents increasingly participate in financial transactions, privacy and trust become critical:

| Challenge | ZK Solution |
|-----------|------------|
| Agent proves it has sufficient funds | Range proof without revealing exact balance |
| Verify payment split correctness | Sum proof without revealing individual amounts |
| Authenticate transaction origin | Knowledge proof without exposing private keys |
| Regulatory compliance | Selective disclosure of transaction properties |

## Architecture

```
┌─────────────┐     ┌──────────────────────┐     ┌─────────────┐
│   Client /   │────▶│  Loka ZK Middleware   │────▶│  Blockchain │
│   AI Agent   │◀────│  (Rust Microservice)  │◀────│  / Verifier │
└─────────────┘     └──────────────────────┘     └─────────────┘
                           │
                    ┌──────┴──────┐
                    │  arkworks   │
                    │  (ZK libs)  │
                    └─────────────┘
```

### Core Components

- **API Layer** (`src/api/`) — REST endpoints built with Actix-web
- **ZK Engine** (`src/zk/`) — Proof generation and verification using arkworks
- **Circuit Library** (`src/zk/circuits.rs`) — Pre-built ZK circuits for common payment operations

## API Reference

### Health Check

```
GET /api/v1/health
```

**Response:**
```json
{
  "status": "healthy",
  "service": "loka-zk-middleware",
  "version": "0.1.0",
  "supported_schemes": ["groth16"],
  "supported_curves": ["bn254"]
}
```

### Generate Square Proof

Proves knowledge of a secret `x` such that `x² = y`, without revealing `x`.

```
POST /api/v1/prove/square
Content-Type: application/json

{
  "secret": 42
}
```

**Response:**
```json
{
  "proof_id": "uuid-v4",
  "proof": "hex-encoded-proof",
  "verification_key": "hex-encoded-vk",
  "public_inputs": ["hex-encoded-y"],
  "scheme": "groth16",
  "curve": "bn254"
}
```

### Generate Sum Proof

Proves knowledge of secret values `a` and `b` such that `a + b = sum`, without revealing `a` or `b`.

```
POST /api/v1/prove/sum
Content-Type: application/json

{
  "a": 100,
  "b": 250
}
```

### Verify Proof

Verifies any Groth16 proof generated by this service.

```
POST /api/v1/verify
Content-Type: application/json

{
  "proof": "hex-encoded-proof",
  "verification_key": "hex-encoded-vk",
  "public_inputs": ["hex-encoded-inputs"]
}
```

**Response:**
```json
{
  "valid": true,
  "scheme": "groth16"
}
```

## Integration with Agentic Payment Flows

### Example: Privacy-Preserving Payment Verification

```
1. Agent A wants to prove it can pay 100 tokens without revealing its balance
2. Agent A calls POST /api/v1/prove/square with its secret balance hash
3. The middleware generates a ZK proof
4. Agent A sends the proof to the payment network
5. The verifier calls POST /api/v1/verify to confirm validity
6. Payment proceeds without any party learning Agent A's actual balance
```

### Example: Confidential Payment Splitting

```
1. Agent needs to split payment between two sub-agents (amounts hidden)
2. Agent calls POST /api/v1/prove/sum with secret amounts a and b
3. Proof demonstrates a + b = total without revealing a or b
4. Each sub-agent receives its share with cryptographic guarantee of correctness
```

## Security Considerations

- **No secret storage**: The middleware never persists private inputs; secrets exist only during proof generation
- **Trusted setup**: Current implementation uses per-request setup. Production deployments should use a ceremony-based trusted setup
- **Transport security**: Always deploy behind TLS in production
- **Input validation**: All inputs are validated before processing
